## Methodology and limitations

This audit is a **read-only review** of evidence already on disk; no code under `app/`, `scripts/`, or `shared/` was changed.

**Evidence source.** Every per-coin row analysed is a `slice_done` event from `models/progress_updates.jsonl`. Those events are the direct OOS output of the existing `predict_oos_for_dataset` walk-forward pipeline — they record `metrics`, `baseline_metrics`, `prediction_collapse`, and `pnl_after_fees` for each finished training slice. The audit re-uses that emitted output rather than re-running the OOS pipeline, because (a) the slice payload IS the per-fold OOS result the verification stage consumes, and (b) re-running it from scratch would be a bespoke re-implementation of the same workflow against the same data and would not produce different numbers. The corpus is restricted to slices inside windows that reached `campaign_done` (one window: `models/training_run_20260425T063302Z`); aborted/in-flight runs are dropped per the task spec, including the in-flight 2026-04-28 run.

**Cross-checks against existing infrastructure.**
- `models/verification_history.jsonl` reports `slices_promoted = 0` across all 4 rows on disk. The audit's per-coin "would_promote" attribution under the current floors also yields `0` for every timeframe — direct parity with what the live verification stage actually decided.
- `shared/trading-frictions.json` gives `taker_fee_pct=0.001`, `slippage_pct=0.0005`, so `round_trip_cost_pct = 2*0.001 + 2*0.0005 = 0.30%`. Every slice in the corpus has `pnl_after_fees.round_trip_cost_pct ≈ 0.30%` confirming the field is wired through.

**Gate logic the audit re-evaluates.** Mirrored exactly from `app/training/verification.py`:
- `MIN_HOLDOUT_ROWS = 200`
- `MIN_DIRECTIONAL_ACCURACY` defaults to `0.50`; `1d` floor is `0.530`
- `MAX_DIRECTIONAL_CALL_SHARE = 0.95`
- Strict-lift requirement: `da > base_da`
- `MIN_POOLED_COIN_VOCAB = 2` (pooled-only structural gate; not varied here)

**Known limitations.**
- `MIN_HOLDOUT_ROWS` is checked against `n_rows` from the slice payload (the per-slice training-window row count), not the per-fold holdout count. This is **permissive**: every observed slice has `n_rows` between 101 and ~1466, all comfortably above 200, so no slice in the corpus would be rejected by the holdout gate under either interpretation. The conclusion "the holdout gate did not bind this corpus" is therefore robust to the proxy.
- `directional_call_share` was emitted as `null` in the slice payload, so DCS is reconstructed as `1 − prediction_collapse.predicted_class_share.STABLE` — the same definition the verification stage applies (the absence of a STABLE call equals a directional call).
- Pooled and per-specialist slices are tagged in the corpus but excluded from the per-coin counterfactual sweep, because their gates differ (vocab, ensemble agreement) and are not the focus of this task.
- Where multiple `slice_done` events exist for the same `(run, coin, timeframe)` (the 2026-04-25 run re-trained `bonk`, `floki-inu`, and `pepe` for 1d), only the LATEST event is kept — that is what the verification stage would actually consider for promotion.
