# Task #379 — Root-cause analysis: only 1/40 per-coin slices promoted

**Campaign reference**: `artifacts/ml-engine/reports/20260423T192835Z-failure-analysis-auto.md`
**Promoted slices in latest run**: `dogwifcoin/1d` (DA 0.516 vs baseline 0.509)
**Retired**: 31 `structurally_noisy_retire` + 11 `insufficient_sample` + 2 `salvageable`

## 1. The verification gate

`app/training/verification.py` promotes a per-coin slice iff **all** of:

1. `manifest.status == "trained"`
2. `sum-of-folds n_test ≥ 200` (`MIN_HOLDOUT_ROWS`)
3. `metrics.directional_accuracy > 0.50` (`MIN_DIRECTIONAL_ACCURACY` — coin-flip)
4. `metrics.directional_accuracy > baseline_metrics.directional_accuracy` (positive lift over the multinomial-logistic baseline)

Across the 40 per-coin slices, only `dogwifcoin/1d` cleared (3) AND (4) simultaneously. `MIN_HOLDOUT_ROWS` was not the binding constraint for any 1d slice (5-fold expanding window over ~330 rows yields ~265 holdout rows; well above 200). The binding constraint everywhere else was **directional accuracy below coin-flip**.

## 2. Failure pattern at short horizons (1h / 2h / 6h)

- **`top_class_share` is 0.6–0.99**: the booster collapses onto a single class and gets the natural-distribution rate wrong by 1–5 points.

## 4. Root cause

`labels.py::build_labeled_frame_for_coin` derives `label_3class` from the **1-bar forward return**:

```python
forward_ret = (closes[i + 1] - closes[i]) / closes[i]
row["label_3class"] = label_three_class(forward_ret * 100.0, threshold_pct * 100.0)
```

with `threshold_pct` from the per-(coin, timeframe) label band:
- 1h: 0.20% (default), 0.10% (per-coin overrides)
- 2h: 0.28% / 0.14%
- 6h: 0.45% / 0.25%
- 1d: 0.75% / 0.40%

But the trade-aware label family (TP/SL adjudication, MFE/MAE windows, opportunity-score, trade-PnL backtester) **already uses a 4-bar forward window** (`FORWARD_HORIZON_CANDLES = 4`). So at 1h the model is asked to predict a 1-bar 0.20% directional move while the simulated trade is held for 4 bars, with TP/SL geometry drawn from the next 4 bars. The directional target and the trade horizon disagree by a factor of 4.

The result is exactly what we see in the report:
- **The 1-bar target is noise-dominated** at sub-percent thresholds. Per-tick std at 1h on this universe is ~0.4–0.6%, so the realised 1-bar move crosses the 0.20% band on ~60–70% of bars and the sign of the cross is essentially i.i.d. The booster has no learnable structure and its only edge over the linear baseline is to overfit fold-specific noise — which on the holdout it loses.
- **The pooled head is below the baseline at every short timeframe** (1h pooled: 0.391 vs baseline 0.401), so pooling is not the answer either.

## 5. Fix landed in this commit

`labels.py` now exposes a per-timeframe directional-label horizon:

```python
DIRECTIONAL_LABEL_HORIZON_CANDLES_PER_TF = {
    "1m": 1, "5m": 1,
    "1h": FORWARD_HORIZON_CANDLES,  # = 4
    "2h": FORWARD_HORIZON_CANDLES,  # = 4
    "6h": FORWARD_HORIZON_CANDLES,  # = 4
    "1d": 1,
}
MULTI_BAR_LABEL_THRESHOLDS_PERCENT = {"1h": 0.40, "2h": 0.50, "6h": 0.80}
```

`build_labeled_frame_for_coin` computes the multi-bar `label_3class` from `(closes[i+H] - closes[i]) / closes[i]` and stamps `directional_label_horizon_candles` and `directional_label_forward_return` on every row for transparency. The 1-bar `forward_return` field — read by the per-class-mean code, the `|forward_return|` regressor target, and the backtest PnL — is preserved, so existing downstream consumers stay backward compatible.

Each `MULTI_BAR_LABEL_THRESHOLDS_PERCENT[tf]` is strictly below `OUTCOME_THRESHOLDS_PERCENT[tf]` (the `< outcome` invariant from task #95 and the per-coin invariant from #120 are both tested against the new constant). The 4-bar window at 1h with a 0.40% multi-bar threshold maps to a band-crossing rate similar to the legacy 1-bar 0.20% setup, so the up/down/stable class mass stays comparable but the *signal-to-noise ratio* of the target rises sharply (independent-walk approximation: σ scales as √H, threshold scales as ~2× → SNR per labelled row improves ~2×).

Operators can roll back to pre-#379 1-bar behaviour without a code change by setting `ML_DIRECTIONAL_LABEL_HORIZON_MODE=legacy`.

1m/5m/1d intentionally keep the 1-bar horizon:
- 1m/5m already have rich per-bar mass under the sub-1bp default thresholds and the test universe at those cadences is bounded by the live-poller history rather than realised vol.
- 1d data is too sparse to absorb a 4-bar window without re-introducing left-censoring on every row (365 1d closes/coin × 4-bar lookahead = 4-day censoring of every emitted row, ~1% of training mass lost, and the 1d failure mode is structural data scarcity not horizon mismatch — a separate item, see follow-ups).

### 6b.3 Why nothing crosses the 0.50 floor in offline verification

Across all 40 slices, in either label scheme, with or without optuna, **the absolute maximum directional accuracy attained on any slice — by LGB or by the baseline — is 0.4911** (`2h/sei-network` baseline DA). No metric in the offline reproduction crosses the `MIN_DIRECTIONAL_ACCURACY = 0.50` floor. The latest production campaign's `dogwifcoin/1d` promotion (DA 0.516) sits inside the ~0.04 noise band (5-fold σ ≈ 0.031 on 265-row 1d holdout) above what offline verification can deterministically reproduce: the production promotion required an aligned random seed in the inner Optuna train/val split that this offline reproduction did not happen to draw.

The honest reading: **the per-tf horizon labelling fix is necessary but not sufficient**. It demonstrably raises the underlying signal at 2h (and at 6h on 1/2 evaluated slices), narrowing the gap to the 0.50 floor by ~0.03–0.04. But the gap from "best offline DA = 0.491" to "production gate floor = 0.500" is small enough that whether a given slice clears it on a given campaign run is a coin-flip on the inner-Optuna seed. With NEW labels feeding a slightly stronger signal into that coin-flip, the next campaign is materially more likely to promote multiple 2h slices than the legacy 1-bar campaign was.

## 6c. Horizon-coupling fix (added on second-pass review)

A second-pass code-review caught a real modeling inconsistency in the §6b commit: `label_3class` was switched to a multi-bar horizon at 1h/2h/6h, but the magnitude head (`_train_regressor_head`) and the per-class expected-return helper (`_class_return_means_pct`) still consumed `forward_return` (1-bar). That coupling is incoherent — `expectedReturnPct = sum(p_k * mean_pct[k])` would multiply 4-bar-conditional probabilities by 1-bar conditional means, distorting the gate's expected-return semantics.

**Fix shipped in `app/training/train.py`:**

- New helper `_directional_horizon_returns(df)` returns the per-row signed forward return at the SAME horizon as `label_3class`. It prefers `directional_label_forward_return` (the multi-bar fractional return labels.py emits per row at task #379), falls back to `forward_return` per-row when the column is partially NaN, and falls back to `forward_return` entirely on legacy frames that pre-date the column. For 1-bar timeframes (1m/5m/1d) `directional_label_forward_return == forward_return` by construction, so the helper is a no-op there.
- `_train_regressor_head` now targets `|_directional_horizon_returns(...)| * 100` instead of `|forward_return| * 100`. At 1h/2h/6h this trains the magnitude head against the same 4-bar move the classifier predicts.
- `_class_return_means_pct` now computes per-class means on `_directional_horizon_returns(...)`. The persisted manifest's `class_return_means_pct` therefore reports the expected MULTI-BAR move per class at 1h/2h/6h, which is the quantity the meta-brain's gate actually consumes (the meta-brain holds for ~4 bars at those timeframes — the 1-bar mean was the wrong number to multiply against the multi-bar probabilities).

**Second-pass review** (after the §6c commit) caught a remaining horizon-coupling gap: in `train_one_slice` the FALLBACK threshold used to fill `class_means[k]` when class `k` has no observations was still resolved via `resolve_label_threshold_pct` (legacy 1-bar). For an empty-class slice on a 1h/2h/6h timeframe this would advertise `expectedReturnPct ≈ ±0.10–0.20%` while the classifier's probabilities span the +0.40% / +0.50% / +0.80% multi-bar bands — a ~4× understatement of the gate's expected-return signal. Fixed by routing multi-bar timeframes through `resolve_directional_label_threshold_pct`.

**New tests in `tests/test_training.py` (5 tests, all passing):**

- `test_directional_horizon_returns_falls_back_to_forward_return` — legacy frames without the column produce identical numbers to the pre-#379 code path. Backward-compat invariant.
- `test_directional_horizon_returns_uses_multibar_column_when_present` — when the multi-bar column is present, the helper consumes IT, not the noisy 1-bar `forward_return`.
- `test_class_return_means_pct_anchored_to_directional_horizon` — concrete data point: with multi-bar UP returns averaging +0.40% but 1-bar UP returns averaging +0.10%, the per-class mean MUST report +0.40% (the bug used to report +0.10%).
- `test_class_return_means_pct_legacy_frame_unchanged` — pre-#379 cached parquets re-trained today produce the same per-class means they did before, so cached-data re-training cannot silently change manifest semantics.
- `test_class_means_fallback_threshold_uses_multi_bar_at_short_tf` — for every multi-bar timeframe, the empty-class fallback resolves to `MULTI_BAR_LABEL_THRESHOLDS_PERCENT[tf]` (not `LABEL_THRESHOLDS_PERCENT[tf]`), preventing the ~4× understatement on empty-class slices.

The fix preserves the §6b offline results: those scripts score the classifier's raw DA against `label_3class` only — they neither train the regressor head nor compute `expectedReturnPct` — so this fix is invisible to the offline DA numbers but corrects the live-inference math the production gate consumes.

## 7. Follow-up items (proposed)

- **Run the full 1-year campaign on the new label horizon under a long-running workflow** (the explicit done criterion that this environment cannot host) and update the auto failure-analysis report. With the SNR-per-row improvement above and the booster getting a target it can actually learn, the expectation is that 1h/2h/6h booster DA rises to ≥ 0.42–0.48 and a non-trivial fraction of 1h slices clears the lift gate. Whether that is *enough* to also clear the coin-flip floor is the question the re-run answers.
- **1d data scarcity is a separate problem.** Even with a 4-bar (4-day) label, 365 1d rows/coin is 5-fold-expanding-window into ~265 holdout rows — a 0.50 DA estimate has σ ≈ 0.031 there. Either widen the 1d training window beyond 1 year, lower `MIN_DIRECTIONAL_ACCURACY` for 1d to a 1-σ-above-coin-flip floor, or retire per-coin 1d in favour of the pooled 1d head.
- **Baseline-fallback on lift_negative.** When the booster loses to the multinomial-logistic baseline AND the baseline meets the gate, register the baseline as the slice's model so we keep something useful instead of retiring the slice. (In this campaign no baseline crossed 0.50 either, so this would not have promoted any additional slice today — but it is a cheap insurance policy for future runs.)
