## Findings

### 1. `price_history` has no native-cadence column today

`lib/db/src/schema/price_history.ts` defines:

```ts
export const priceHistoryTable = pgTable("price_history", {
  id: serial("id").primaryKey(),
  coinId: text("coin_id").notNull(),
  price: real("price").notNull(),
  timestamp: timestamp("timestamp", { withTimezone: true }).notNull().defaultNow(),
  isSynthetic: boolean("is_synthetic").notNull().default(false),
  regime: text("regime"),
});
```

There is **no** `timeframe`, `cadence`, `interval`, `source_cadence_ms`, or even `source` column. Every row is implicitly assumed to be a poll-cadence tick from the live monitor (`market-signals-poller.ts`, `monitor.ts`). Adjacent tables surveyed (`market_signals`, `model_predictions`, `predictions`, `prediction_journal`) all carry `timeframe` for their own purposes but none gates `price_history` writes by it.

### 2. Today the table is in fact uniform-cadence (and contamination is NOT yet occurring)

Static audit of all writers to `price_history`:

| Writer | File:Line | What it writes | Cadence |
|---|---|---|---|
| Live coin poller | `artifacts/api-server/src/lib/monitor.ts:666` | One row per poll cycle | ~live-poll cadence (60s) |
| Historical seeder | `artifacts/api-server/src/lib/monitor.ts:275` | 24 hourly synthetic rows per CMC daily candle | hourly, **`is_synthetic=true`** |
| (#306 backfill modules) | `artifacts/api-server/src/lib/history-backfill/*.ts` | **Not yet present in the repo** — task #306 has not landed | n/a (planned: 1d, 1h) |

The reader filters synthetic rows at SQL (`fetch_real_ticks` in `artifacts/ml-engine/app/db.py:108`):

```sql
WHERE coin_id = $1
  AND (is_synthetic IS NULL OR is_synthetic = false)
  ...
```

So today the trainer sees only live-poll ticks. No daily/hourly bars are present in the real-tick stream. **Therefore today's red verification report is not a casualty of cadence contamination — every slice was built from uniform-cadence ticks.** The gate failure has a different root cause; see the failure-analysis report.

### 3. The risk is forward-looking and load-bearing for #306

The moment #306 lands `history-backfill/coinmarketcap.ts` and `history-backfill/okx.ts`, real (`is_synthetic=false`) **daily** rows from CMC and **hourly** rows from OKX will be written to the same `price_history` table with no marker that distinguishes them from poll-tick rows. From that point on:

- `fetch_real_ticks(coin, lookback)` returns a mixed stream — poll ticks *and* daily/hourly bars, all chronologically interleaved.
- `resample_to_candles(ticks, bucket_ms)` (in `artifacts/ml-engine/app/features.py:35`) walks the stream, assigns each row to a bucket by `floor(ts / bucket_ms) * bucket_ms`, and keeps the **last price in the bucket** as the candle close.
- For any target bucket smaller than the coarsest contributing native cadence, this is **silent corruption**: a 5m bucket whose only contributor in some hour-aligned window happens to be a single CMC daily row will close at that daily close — a value that has no defined relationship to the true 5-minute bar.
- For target buckets larger than the coarsest native cadence (e.g. 1d target with a daily input row), the result is technically correct but the trainer will mistake one row per day for a 365-row dataset of distinct candles, when it is in fact the input bar itself.

There is no current code path that detects this. The leakage audit (`audit_leakage` in `labels.py`) is point-in-time-only; it does not check the cadence of underlying rows.

The verification gate cannot do its job once contamination begins: lift numbers will reflect a mixture of real bar dynamics and resampler artefacts, and individual coins may pass or fail the gate purely depending on which native cadence dominates a given window.

## Proposed migration

We pick **option (b) from the task brief: a separate `price_candles` table** keyed by `(coin_id, timeframe, bucket_start)`. `price_history` stays for raw ticks only. Justification:

1. **Cleanest invariant.** `price_history` row → tick. `price_candles` row → fully-aggregated bar at a known timeframe. The trainer's read paths split cleanly: tick consumers go to `price_history`, bar consumers go to `price_candles`. There is no mixed-cadence query path that *could* return an interleaved stream.
2. **No data loss on existing rows.** Every existing row in `price_history` is a tick; no migration of historical data is required, only a forward-going write split.
3. **Per-bar uniqueness is enforced at the index level.** `(coin_id, timeframe, bucket_start)` is a partial unique index, so a duplicate or conflicting daily row from two sources is rejected at insert time, not silently averaged in by the resampler.
4. **Cross-source attribution is straightforward.** A `source` column on `price_candles` (`"cmc" | "okx" | "coincap"`) lets the cross-source validation step in #306 reject mismatched days deterministically rather than picking the last-writer's price.

Compared to (a) `timeframe` enum on `price_history`: option (a) would force every read path that today expects ticks to add `WHERE timeframe = 'tick'`, and any miss is a silent contamination bug — exactly the failure mode we are trying to make impossible. Option (b) makes the contamination physically impossible: a tick consumer cannot accidentally read a daily bar because they live in different tables.

Compared to (c) `source_cadence_ms` + a hard rule in `resample_to_candles`: option (c) leaves the data interleaved at rest and depends on a runtime check at every read site to keep the invariant. We prefer at-rest separation.

### DDL sketch (Drizzle)

```ts
// lib/db/src/schema/price_candles.ts (new)
import { pgTable, text, real, timestamp, uniqueIndex, index } from "drizzle-orm/pg-core";

export const priceCandlesTable = pgTable(
  "price_candles",
  {
    coinId: text("coin_id").notNull(),
    timeframe: text("timeframe").notNull(), // "1m" | "5m" | "1h" | "2h" | "6h" | "1d"
    bucketStart: timestamp("bucket_start", { withTimezone: true }).notNull(),
    open: real("open").notNull(),
    high: real("high").notNull(),
    low: real("low").notNull(),
    close: real("close").notNull(),
    volume: real("volume"),
    source: text("source").notNull(), // "cmc" | "okx" | "coincap"
    createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
  },
  (t) => ({
    pk: uniqueIndex("price_candles_pk_idx").on(t.coinId, t.timeframe, t.bucketStart),
    byCoinTf: index("price_candles_coin_tf_idx").on(t.coinId, t.timeframe, t.bucketStart),
  }),
);
```

`price_history` keeps its current schema (raw ticks only) — no migration, no backfill of existing rows.

### Read-side changes

- `artifacts/ml-engine/app/db.py`
  - `fetch_real_ticks(coin, lookback_ms)` — unchanged (still reads `price_history`, still tick stream).
  - **New:** `fetch_real_candles(coin, timeframe, lookback_ms)` — reads `price_candles` for the requested timeframe, returns `list[(bucket_start, close)]` already sorted.
- `artifacts/ml-engine/app/training/labels.py`
  - `build_labeled_frame_for_coin(coin, timeframe, ...)` reads candles for any timeframe ≥ 5m via `fetch_real_candles(coin, timeframe, ...)` and **only** falls back to `resample_to_candles` over ticks when no candles are present for that (coin, timeframe). When fall-back happens, the slice's provenance records `bars_source="resampled_ticks"` so the failure-analysis report can split slices by source.
  - The 1m slice is the special case: 1m bars are *only* ever resampled from ticks. The fetcher never reads 1m candles from `price_candles` (no source provides genuine 1m bars in our universe).
- `artifacts/ml-engine/app/features.py`
  - `resample_to_candles` learns a `min_input_cadence_ms` argument (defaults to 0). When set, it asserts every input row's inter-arrival gap is `≤ min_input_cadence_ms` and raises a `CadenceMismatchError` if not. The trainer passes the bucket width as the cap so a daily row can never feed a 5m or 1h bucket.
- `artifacts/ml-engine/app/training/train.py`
  - Per-slice provenance gains `bars_source` (`"resampled_ticks" | "candles"`) and `bars_native_cadence_ms`. The trainer **refuses to train** any slice whose source is `candles` mixed with resampled-ticks for the same target timeframe (the new `fetch_real_candles` query is single-cadence by construction; this guard exists to catch any future regression).

### Write-side changes (deferred to #306 follow-up)

- `artifacts/api-server/src/lib/history-backfill/coinmarketcap.ts` writes 1d bars to `price_candles` with `source="cmc"`.
- `artifacts/api-server/src/lib/history-backfill/okx.ts` writes 1h (and any other native cadence OKX returns) bars to `price_candles` with `source="okx"`.
- The cross-source validation step from #306 S001 compares overlapping `(coin, timeframe, bucket_start)` rows across sources before inserting, rejects mismatched days, and never writes both sides of a conflict.

The actual backfill modules and the migration itself are **not landed in this task** — task #315 is diagnosis-and-plan only. The migration ships as the first step of the repair plan in `20260423T000000Z-failure-analysis.md`.

## Tests added in this task

The cadence-correctness tests live at:

- `artifacts/ml-engine/tests/test_cadence_correctness.py` — three real behavior assertions that fail today and turn green when the schema fix lands. Each test exercises an actual code path with crafted inputs:
  1. `test_daily_rows_are_not_silently_merged_into_5m_bars` — feeds two daily-cadence rows into `resample_to_candles` with `bucket_ms=300_000` and asserts `CadenceMismatchError` is raised. Today's code accepts the call and returns two 5m closes; post-fix it raises.
  2. `test_resample_quarantines_coarser_rows_within_a_bucket` — feeds four 30s ticks at $100 plus a daily contaminant at $999 in the same 5m bucket; asserts the bucket close is $100 (not $999). Today's last-tick-wins behavior returns $999; post-fix the daily row is quarantined.
  3. `test_trainer_provenance_records_native_cadence_and_refuses_mixed` — asserts every per-slice manifest carries both `bars_by_native_cadence` and `cadence_mixed`, AND that `app.training.verification.manifest_blocks_promotion_for_cadence_mix` exists and returns True for a mixed-and-unmitigated manifest. Today no manifest carries either field and the helper does not exist; post-fix both are populated and the helper enforces refusal.

- `artifacts/api-server/test/price-candles-uniqueness.test.ts` — three node-test assertions over the Drizzle schema package: (a) `priceCandlesTable` is exported, (b) it carries the `(coin_id, timeframe, bucket_start, source)` columns, (c) `priceHistoryTable` must NOT gain a `timeframe` column. (a) and (b) fail today; (c) currently passes as a regression guard.

The assertions are hard (not `xfail`/`it.todo`) so they fail loudly today, surface in CI, and stay green once the fix lands. Removing or weakening any guarantee will re-fail the suite.

## Bottom line

- **Today's red verification report is not contamination.** It's a different root cause (label-band mismatch + insufficient sample at coarse timeframes + 1d slice has zero rows because we're resampling 60s ticks into 1d buckets). See `20260423T000000Z-failure-analysis.md`.
- **The schema fix is non-negotiable before #306 lands.** It is preventive work — once daily/hourly real bars start flowing into `price_history`, every red slice in the next gate run becomes ambiguous (was it the model or contamination?) and the gate stops being trustworthy as a signal.
