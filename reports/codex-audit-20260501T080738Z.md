## 2. Sandbox / approval mode (and why)

The task spec asks for "most read-only mode". Codex's nominal `--sandbox read-only`
mode wraps every shell command in `bwrap` (bubblewrap), and the vendored `bwrap`
shipped with `@openai/codex@0.125.0` requires unprivileged user namespaces, which
the Replit container kernel does not allow:

```
bwrap: setting up uid map: Permission denied
namespace_helper: kernel does not allow non-privileged user namespaces
```

In that mode Codex therefore could not execute even a `cat` and produced an
empty audit. Because the Replit container is itself the external sandbox for
this run (no production secrets mounted, repo working tree only, network limited),
Codex was launched with:

```
codex exec --dangerously-bypass-approvals-and-sandbox \
           --color never \
           -m gpt-5.1-codex-mini \
           -o .local/codex_run/last_message.md \
           - < .local/codex_audit_prompt.txt
```

The "no edits / no deletions" guarantee was therefore enforced **out of band**,
not by Codex's own sandbox:

1. The audit prompt itself instructs the agent: *"Do not modify files. Do not
   delete files. Do not apply fixes. Produce an audit report only."*
2. Post-hoc `git diff --name-only HEAD` was checked ŌĆö see ┬¦3.
3. The Codex session JSONL transcript was scanned for any `apply_patch` /
   `write_file` / `update_plan` events that would indicate a write attempt ŌĆö see ┬¦3.
